Got it. Here's the architecture for a token-efficient code scanning + reorganizing agent:

The core principle: grep/AST/regex does the reading, Claude only makes decisions.

Phase 1 — Zero-token scanning (pure tools)

The agent uses shell commands and language-native parsers to build a structural map without touching the Claude API at all. For Python it uses ast.parse() to extract every class, function, import, and dependency graph. For JS/TS it uses acorn or esprima. For any file type, grep extracts function signatures, class names, import statements, and section markers. wc -l gives size. The output is a compact JSON summary — typically 200–500 tokens worth of structured metadata, not the raw source.

Phase 2 — Single Haiku call (~500 tokens in, ~300 out)

The structural JSON gets sent to Claude Haiku (cheapest model, $0.25/$1.25 per million tokens) with a prompt like: "Here is the dependency graph and function inventory. Identify: (1) duplicated logic, (2) logical groupings, (3) recommended file split boundaries." One call. One response. Total cost: fractions of a cent per file.

Phase 3 — Rule-based reorganization (zero tokens)

The agent applies Haiku's decisions mechanically — moving functions, injecting section comments, extracting shared utilities, splitting files — using pure Python string manipulation and AST transformation. No more Claude calls.

Why this beats naïve approaches

Sending a full 3,700-line file to Claude Sonnet costs ~50,000 tokens (~₹35). The agent approach costs ~800 tokens total (~₹0.05) and handles arbitrarily large codebases because it never loads full source into context — only the map.

What it reorganizes

At the file level: groups related functions together, adds section headers, sorts imports. At the project level: extracts duplicated helpers into a shared module (exactly the problem your four build scripts have — all four define the same fill(), font(), border() helpers independently). At the architecture level: suggests file split boundaries based on dependency clustering — functions that only call each other form natural module candidates.

The one limitation to be honest about

It can't rename things intelligently or understand intent — only structure. For semantic refactoring ("this function does too many things, split it") you still need a full model. But for structural cleanup, deduplication, and reorganization, the zero-token scan + Haiku decision loop handles 90% of real-world cases at essentially zero cost.
